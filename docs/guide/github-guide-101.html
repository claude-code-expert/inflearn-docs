00버전관리와 GitHub개념

버전관리 시스템(VCS, Version Control System)은 파일의 변경 이력을 보관·저장하는 시스템입니다. 흔히 버전 관리 또는 형상 관리라고 부릅니다.

과거에는 CVS·SVN처럼 중앙 서버에 변경 기록을 두고 네트워크로 내려받는 방식이 일반적이었으나, 저장소가 커질수록 느려지고 네트워크 단절에 취약했습니다. 이를 보완하기 위해 각자의 로컬에 저장소 전체를 복제하는 분산형 버전관리(DVCS, Distributed VCS)가 도입되었고, 그 대표가 바로 Git / GitHub입니다.

GitHub은 로컬 복제 기반으로 가볍고 빠르며, 대형 오픈소스 프로젝트와 스타 개발자들이 모이면서 사실상 표준이 되었습니다. 기본 사용은 무료이며, 기업용 솔루션·프로젝트 관리·AI 코드 보조(GitHub Copilot 등) 제품군도 함께 제공합니다. 2018년 Microsoft에 인수되어 운영 중입니다.

GitHub을 다루는 방법

01GitHub 가입 & 첫 리포지토리실습

GitHub 메인 화면에서 Sign up 버튼을 눌러 계정을 만들 수 있습니다. 상단 메뉴 Product → Documentation이나 GitHub Skills에서 공식 학습 자료도 볼 수 있습니다 — 영문 독해가 익숙하다면 공식 Doc·Skill 문서를 먼저 권합니다.

1) 계정 만들기

이메일로 로그인 계정을 만듭니다. 패스워드를 입력한 뒤 GitHub에서 사용할 유저명을 정합니다.

유저명은 URL 경로가 됩니다

유저명은 github.com/<유저명>처럼 주소 경로로 그대로 쓰이므로 신중하게 정하세요. 나중에 변경은 가능하지만 기존 링크가 깨질 수 있습니다.

업데이트 메일 수신 여부를 묻는 칸이 나옵니다(받으면 y, 아니면 n). 이어서 스팸 방지용 간단한 퍼즐을 풀고, 가입을 위한 활성화 코드 입력 단계로 넘어갑니다.

가입이 정상 진행되면 이메일로 launch code(활성화 코드)가 전송됩니다. 이 코드를 화면에 입력하면 가입이 완료되고, 간단한 사용자 설문(사용 인원, 학생/교사 여부, 사용하려는 기능 등)이 이어집니다.

설문에 따라 추가 옵션이 나오는데, Continue for free를 선택하면 개인 대시보드로 이동합니다.

2) 첫 리포지토리(저장소) 만들기

처음에는 저장소(Repository)가 없으므로 빈 화면이 나옵니다. Create repository를 눌러 만들어 봅시다.

리포지토리명과 설명을 입력하고, Public(공개)과 Add a README file 정도만 선택해 생성하면 저장소가 만들어집니다. 👏 이제 GitHub에 첫 깃발을 꽂았습니다.

GitHub 주소는 github.com/<USER-ID>/<저장소명> 형식입니다. 예: https://github.com/USER-ID/git-sample.

Code 버튼을 누르면 클론에 쓰는 주소가 나옵니다. HTTPS, SSH(로컬에 public key 등록 필요), CLI 주소를 선택할 수 있고, 소스를 zip으로 내려받을 수도 있습니다.

README의 펜 아이콘을 눌러 Markdown으로 내용을 적고, Commit changes에 무엇을 바꿨는지 기록한 뒤 커밋하면 변경이 화면과 파일 리스트에 반영됩니다. 기본적인 첫 스텝은 여기까지입니다.

02내 PC에 Git 설치하기 (macOS)실습

윈도우는 git-scm.com/download/win에서 인스톨러를 받아 설치하면 끝입니다. 맥은 최초 실행 시 몇 가지 사전 작업이 필요합니다.

1) Git 설치 여부 확인 & Xcode Command Line Tools

git --version

버전이 출력되면 설치된 것이고, 아니면 추가 설치가 필요합니다. 맥에서는 보통 저장소를 clone해서 작업하는데, Git이 없으면 clone 시 Xcode Command Line Tools 설치 안내가 뜹니다.

git clone https://github.com/<user-name>/git-sample.git
xcode-select: note: no developer tools were found...
requesting install. Choose an option in the dialog to download
the command line developer tools.

2) Homebrew 설치하기

brew.sh의 공식 명령을 터미널에 붙여넣습니다.

/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

Apple Silicon(M1·M2·M3 등)에서 brew: command not found가 날 때

Apple Silicon 맥은 Homebrew가 /opt/homebrew에 설치됩니다(Intel은 /usr/local). 설치 직후 셸이 brew 경로를 모르면 zsh: command not found: brew가 납니다. 아래처럼 ~/.zprofile에 환경 설정을 추가하면 됩니다.

# <MAC_USER_ID>는 본인 맥 계정명. 모르면 터미널에 pwd 입력해 확인
echo 'eval "$(/opt/homebrew/bin/brew shellenv)"' >> /Users/<MAC_USER_ID>/.zprofile
eval "$(/opt/homebrew/bin/brew shellenv)"   # 현재 셸에 즉시 적용

brew help   # 도움말이 출력되면 정상 설치
참고

최신 Homebrew 설치 스크립트는 설치가 끝나면 위 ~/.zprofile 추가 명령을 "Next steps"로 직접 출력해 줍니다. 출력된 두 줄을 그대로 복사·실행하면 됩니다. .zprofile이 이미 있으면 eval "$(/opt/homebrew/bin/brew shellenv)" 한 줄만 추가하세요.

3) Git 설치

brew install git

설치가 끝났다면 작업용 디렉터리를 만들고 그 안에서 저장소를 클론해 봅시다.

mkdir git-workspace
cd git-workspace
git clone https://github.com/<user_name>/git-sample.git

03첫 커밋과 푸시 — 인증의 변화실습

클론한 git-sample 디렉터리로 이동해 상태를 확인합니다.

cd git-sample
git status   # 변경사항이 없으면 "nothing to commit" 출력

파일을 만들거나 수정한 뒤 커밋해 봅시다. README가 없다면 새로 만들고, 이미 있다면 README-TEST.md 같은 별도 파일로 테스트해도 됩니다.

touch README.md       # 파일이 없을 때 새로 생성
# 내용을 아무거나 입력해 저장한 뒤
git add README.md     # Git이 추적(tracking)하도록 추가
git commit -m "첫 번째 커밋입니다."

최초 사용자 정보 설정

이름·이메일을 한 번도 설정하지 않았다면 커밋 시 다음과 같은 안내가 나옵니다. 자동 추정값 대신 본인 정보를 명시적으로 설정해 둡시다.

git config --global --edit   # 글로벌 설정 파일을 vi로 연다
vi가 익숙하지 않다면

수정 후 ESC → :wq! → Enter로 저장·종료합니다. vi가 부담스럽다면 기본 편집기를 바꿀 수 있습니다 — 예: git config --global core.editor "code --wait" (VS Code). 또는 편집 없이 한 줄로: git config --global user.name "이름", git config --global user.email "you@example.com".

push — 그리고 "비밀번호"의 진실

git push origin main   # 원격 main 브랜치에 변경 반영

여기서 GitHub 로그인 암호를 넣으면 실패합니다. 보안 정책이 바뀌어 암호 인증이 폐지되었기 때문입니다.

remote: Support for password authentication was removed on August 13, 2021.
remote: Please see https://docs.github.com/get-started/getting-started-with-git/
        about-remote-repositories for currently recommended modes of authentication.
fatal: Authentication failed for 'https://github.com/<user_name>/git-sample.git/'
암호 인증은 2021년 8월 13일에 폐지되었습니다

HTTPS 방식 Git 작업에서는 비밀번호 대신 Personal Access Token(PAT)을 사용해야 합니다. (또는 처음 push할 때 자동으로 뜨는 브라우저 OAuth 로그인을 따라가도 됩니다.) PAT는 HTTPS URL 작업에만 쓰입니다.

인증 방식주소 형태준비물
HTTPS + 토큰https://github.com/<id>/<repo>.gitPersonal Access Token 발급 (다음 절)
SSH 키git@github.com:<id>/<repo>.git로컬에 SSH 키 생성 후 Settings에 public key 등록

토큰이든 SSH 키든 편한 방식을 고르면 됩니다. 이 문서는 HTTPS + 토큰을 기준으로 진행합니다.

04Personal Access Token 발급실습

로그인 후 우측 상단 계정 아이콘 → Settings로 들어갑니다. 좌측 메뉴 맨 아래 Developer settings → Personal access tokens에서 토큰을 발급합니다.

두 가지 토큰 — Fine-grained vs Classic

종류특징권장 상황
Fine-grained
(권장)		 특정 저장소·특정 권한(Contents, Pull requests 등)만 콕 집어 부여. 필수 만료일. 대부분의 경우 — GitHub 공식 권장
Classic repo 같은 넓은 스코프 단위. 발급이 단순하지만 권한이 광범위. fine-grained가 아직 지원하지 않는 일부 기능이 필요할 때
최신 변경 — Fine-grained 토큰이 정식(GA)이 되었습니다

Fine-grained PAT는 오랫동안 베타였으나 2025년 3월 18일 정식 출시(GA)되었고, GitHub은 이제 가능하면 fine-grained 토큰 사용을 공식 권장합니다. 따라서 새로 발급한다면 fine-grained를 기본으로 두고, 저장소 하나에 Contents: Read and write 권한만 부여하면 충분합니다. (Classic 토큰도 여전히 동작하며, 폐지되지는 않았습니다.)

Classic 토큰으로 빠르게 발급하기

개인 저장소 하나만 다룬다면 classic도 간단합니다. Generate new token을 누르고,

  • Note — 토큰 용도를 적어 둡니다(터미널용/개발툴용 등 헷갈리지 않게).
  • Expiration — 만료일을 설정합니다(보안상 무기한은 비권장).
  • Scope — repo를 선택(필요 시 추가).
토큰은 생성 직후 한 번만 보입니다

화면을 벗어나면 다시 볼 수 없습니다. 잊으면 재발급해야 하니, 안전한 곳(비밀번호 관리자 등)에 즉시 복사·보관하세요. 평문으로 리포지토리나 dotfiles에 올리지 마세요.

이제 다시 push하면 Username에는 GitHub 아이디, Password에는 방금 복사한 토큰을 넣습니다.

05push 전 3단계 — fetch · pull · push개념

초심자가 가장 많이 겪는 오류가 동기화 누락입니다. 둘 이상이 작업하면 원격 저장소가 내 커밋보다 앞서 있을 수 있습니다. push 전 아래 순서를 기억하세요.

① 원격 정보 가져오기 — fetch

git fetch origin   # 원격 변경 "정보"만 가져온다 (아직 병합 안 함)

② 로컬과 병합하기 — pull

git pull origin main   # 가져온 변경을 내 로컬 브랜치에 병합

③ 병합본을 커밋하고 올리기 — push

git add README-TEST.md
git commit -m "커밋 메시지"
git push origin main
언제 fetch·pull을 생략해도 되나

방금 clone한 직후처럼 원격에 변경이 없을 때는 바로 push해도 됩니다. 그러나 협업 중이거나 작업에 시간이 걸렸다면 push 전 반드시 fetch → pull로 동기화하세요.

토큰을 잘못 넣었거나 권한이 부족하면 403 에러가 납니다.

remote: Permission to <id>/git-sample.git denied to <id>.
fatal: unable to access '...git-sample.git/': The requested URL returned error: 403

정상적으로 push되면 다음과 같이 출력됩니다.

$ git push origin main
...
To https://github.com/<user_name>/git-sample.git
   9c7783c..cfb96d3  main -> main

06브랜치로 작업하고 PR로 합치기실습

여러 작업자가 같은 저장소에 서로 다른 기능을 반영하면 충돌(conflict)이 잦아집니다. 브랜치는 원본과 동일한 버전에서 분기한 독립 저장소로, 각자 따로 작업한 뒤 합치면 충돌을 크게 줄일 수 있습니다. Git은 브랜치 생성·이동·병합이 가볍고 빠른 것이 강점입니다.

브랜치 만들고 이동하기

git branch develop        # develop 브랜치 생성
git switch develop        # 해당 브랜치로 이동 (구: git checkout develop)

git switch -c develop     # 생성 + 이동을 한 번에 (구: git checkout -b develop)
checkout이 switch · restore로 나뉘었습니다

Git 2.23(2019)부터 역할이 과하게 많던 git checkout이 브랜치 전환용 git switch와 파일 복원용 git restore로 분리되었습니다. checkout도 여전히 동작하지만, 새로 배운다면 의미가 명확한 switch/restore를 권장합니다.

브랜치에서 작업 → 원격에 push

브랜치로 이동해 파일을 추가합니다(여기서는 index.html).

git add index.html
git commit -m "develop 브랜치에 html 파일을 추가"
git push -u origin develop   # -u: 원격 develop과 추적 연결까지 설정

push 결과에 PR 생성 안내 URL이 함께 출력되고, GitHub 메인 화면의 브랜치 목록에 develop이 보입니다.

Pull Request(PR)로 병합 요청

새 브랜치를 push하면 Compare & pull request 버튼이 생깁니다. 이를 통해 메인 브랜치와 변경 내용을 비교·논의·리뷰하고 병합합니다.

main 브랜치는 항상 완성본으로

실무에서는 동료 리뷰를 거쳐 PR을 Merge합니다. main 브랜치 소스는 불완전한 상태로 두지 않는 편이 좋습니다. (예제는 혼자 작업이므로 Self PR → Self Merge로 진행합니다.)

Confirm 후에도 커밋을 되돌릴(Revert) 수 있고, 병합에 쓴 브랜치를 삭제할 수도 있습니다(보통 히스토리 파악을 위해 일정 기간 보관).

07인증 에러(403) 해결 — 키체인트러블슈팅

이미 Git이 설치된 상태에서 계정을 추가·변경하면, 예전에 저장된 자격증명 때문에 토큰이 인식되지 않아 403이 날 수 있습니다. macOS는 자격증명을 키체인(Keychain)에 보관하므로, 여기서 값을 갱신하거나 삭제하면 됩니다.

터미널에서 한 번에 지우기

키체인 GUI 대신 다음 명령으로 저장된 GitHub 자격증명을 비울 수도 있습니다.

git credential-osxkeychain erase
host=github.com
protocol=https
(빈 줄에서 Enter 두 번)

이후 다시 push하면 Username·토큰을 새로 묻습니다.

08Git 명령어 모음 (치트시트)레퍼런스

1) 사용자 글로벌 세팅

git config --global user.name "UserName"          # 유저명
git config --global user.email you@example.com    # 유저 이메일

설정 후 clone·pull 시 Username·Password를 물으면, 패스워드 자리에 토큰을 넣습니다.

(1) 자격증명 캐싱

git config --global credential.helper cache       # 토큰을 일정 시간 캐싱
git config --global --unset credential.helper     # 캐싱 해제
# macOS는 osxkeychain, Windows는 manager가 기본 helper로 더 편리

(2) 로컬 저장소 .git/config 직접 수정

[remote "origin"]
    url = https://<user-id>:<token>@github.com/<계정명>/<저장소명>.git
    fetch = +refs/heads/*:refs/remotes/origin/*
[branch "main"]
    remote = origin
    merge = refs/heads/main
URL에 토큰을 박는 방식은 주의

위처럼 URL에 토큰을 넣으면 매번 입력은 줄지만, 토큰이 평문으로 파일에 남습니다. 가능하면 credential helper(키체인/manager)에 맡기는 편이 안전합니다.

2) 최초 설정을 위한 기본 명령

mkdir project        # 로컬 프로젝트 폴더 만들기
cd project
git init             # 이 폴더를 저장소로 초기화
git add README.md    # 파일 추가(스테이징)
git status           # 상태 확인
git commit -m "첫 번째 커밋"

3) 브랜치

git branch newBranch        # 브랜치 생성
git branch                  # 로컬 브랜치 목록
git branch -a               # 로컬+리모트 전체 (-r 리모트, -l 로컬)
git branch -M newName       # 브랜치명 변경 (master → main 컨벤션 변경)
git branch -D localBranch   # 로컬 브랜치 강제 삭제

git switch newBranch        # 브랜치 이동 (구: git checkout)
git switch -c newBranch     # 생성 + 이동 (구: git checkout -b)
기본 브랜치 이름: master → main

GitHub은 2020년 10월부터 새 저장소의 기본 브랜치를 main으로 변경했습니다. (Git 코어 자체는 아직 기본값이 master지만, 차기 메이저에서 main으로 전환될 예정입니다.) 아래 명령어 예시의 master는 환경에 따라 main으로 읽으세요.

4) 원격 저장소

git remote add origin https://github.com/<user>/<repo>.git   # origin 별칭으로 원격 추가
git remote -v                                # 등록된 원격 확인
git remote remove origin                     # 원격 삭제
git remote set-url origin https://.../<repo>.git   # 원격 URL 변경

git fetch --all                              # 모든 원격 정보 가져오기
git pull --all                               # 모든 브랜치 병합

5) 병합 — merge · rebase

git switch main
git merge newBranch          # newBranch를 main에 병합 (fast-forward 가능)

# rebase: newBranch를 main 위로 재배치
git switch newBranch
git rebase main
# 충돌 시: 파일 수정 → git add → 아래로 계속
git add file.txt
git rebase --continue
git rebase --abort           # rebase 취소

git merge --squash newBranch # 여러 커밋을 하나로 묶어 병합

6) 되돌리기 — reset · revert

명령효과언제
git reset HEAD^최종 커밋 취소, 워킹트리 보존커밋했지만 push 전
git reset HEAD~2마지막 2개 커밋 취소, 워킹트리 보존여러 커밋 되돌릴 때
git reset --hard HEAD~2마지막 2개 커밋 + 변경까지 완전 폐기변경 자체를 버릴 때 (주의)
git reset --hard ORIG_HEAD직전 reset/merge 이전 상태로 복구실수로 reset/merge 했을 때
git revert HEAD변경을 취소하는 새 커밋 생성이미 push한 커밋을 되돌릴 때

7) cherry-pick · 로그 · 커밋 수정

git log                      # 커밋 로그 확인
git switch main
git cherry-pick 99daed2      # 특정 커밋만 골라 현재 브랜치로 복사
git commit --amend           # 직전 커밋 메시지/내용 수정

8) 자주 만나는 에러

refusing to merge unrelated histories

두 브랜치의 커밋 이력이 서로 독립적으로 생성되어 연관이 없을 때 발생합니다. 명시적으로 허용 옵션을 줍니다.

git pull origin main --allow-unrelated-histories
# 또는
git merge --allow-unrelated-histories main

src refspec main does not match any

push하려는 브랜치가 로컬/원격에 없을 때입니다. 브랜치 존재를 확인하고, 없으면 초기화부터 다시 진행합니다.

git init
git add README.md
git commit -m "first commit"
git branch -M main
git remote add origin https://github.com/<USER-ID>/<repo>.git
git push -u origin main

9) 기존 로컬 소스를 새 원격에 연결할 때

git init
git remote -v                       # 원격 확인
git remote remove origin            # 기존 원격 정리(있을 때)
git remote add origin https://github.com/<MyAccount>/<repo>.git
git add .
git commit -m "init"
git pull origin main --rebase       # 로컬 변경을 원격 최신 위로 재배치
git push origin main

09브랜치 관리 전략 — Git Flow개념

브랜치 운영 전략은 팀마다 다르지만, 가장 널리 알려진 것이 Git Flow(정식: Git Flow branching model)입니다. 로컬·개발·테스트·스테이징·운영 같은 Phase 단위로 머지 대상 브랜치를 두어 소스 흐름을 통제하는 것이 핵심입니다.

브랜치역할
master / mainProduction에 배포되는 메인. 항상 최신 배포 상태 유지. (과거 master → 현재 main)
develop개발 통합 브랜치. 각자의 feature를 모아 테스트 환경에 배포.
feature기능 단위 브랜치. 완료되면 develop에 머지.
releasemaster에서 파생한 실배포본(QA·카나리 검수 대상).
hotfix배포 후 긴급 버그 패치용. master에서 파생.

feature 브랜치 작업 흐름

보통 develop 기준으로 네이밍 룰을 적용해 브랜치를 만듭니다. Jira를 쓴다면 이슈 번호와 결합할 수 있습니다.

git switch develop
git pull
git switch -c feature/ticket-<번호>-<feature명>
# 작업 완료 → develop에서 fetch & pull 후 merge (daily merge) → push → PR(작업 → develop)

develop에서 테스트 Phase 배포 후 검증 → master로 PR 병합 → Stage 서버 배포·검증 → 배포본 태그 생성.

git switch master
git tag release/REL-2025-06-01-1330       # prefix + 날짜·시간 태그
git push origin release/REL-2025-06-01-1330
# hotfix: master에서 파생 → 작업 → PR(hotfix → master). 태그는 항상 master에서 생성
실무 권장

develop과 feature의 버전 차가 벌어지면 병합이 고통스러워집니다. Daily merge로 꾸준히 동기화하세요. 또한 Git Flow는 공유 브랜치에서 이력을 헝클 수 있는 rebase를 권장하지 않습니다.

2026년 시점의 균형 잡힌 시각 — Git Flow가 정답은 아닙니다

Git Flow를 고안한 Vincent Driessen은 2020년, 원문에 직접 주석을 달아 "지속적으로 배포되는 웹 앱이라면 Git Flow보다 GitHub Flow 같은 단순한 모델을 권한다"고 밝혔습니다. 오늘날 CI/CD 환경의 사실상 표준은 트렁크 기반 개발(Trunk-Based Development)입니다. 릴리스 주기가 분명한 패키지·앱에는 Git Flow가 여전히 유효하지만, 빠른 배포가 핵심이라면 GitHub Flow나 트렁크 기반을 먼저 검토하세요. 전략은 팀의 배포 방식에 맞춰 고르는 것입니다.
