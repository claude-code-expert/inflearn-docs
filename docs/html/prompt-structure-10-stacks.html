00결론 먼저 — 왜 "구조"인가개념

좋은 프롬프트는 멋진 문장이 아니라 잘 쌓은 블록입니다. 정체성(누구) → 자료(무엇을 참조) → 규칙(어떻게) → 예시(이렇게) → 즉시 요청(지금 이것) → 출력형식(이 형태로) 순서로 쌓으면, 모델의 출력이 매번 비슷하게, 예측 가능하게 나옵니다.

핵심 원칙 두 가지:

  • 단일 목적 — 한 블록은 한 가지 일만 한다. 규칙 블록에 데이터를 섞지 않는다.
  • 경계 구분 — 각 자료·예시를 <guide>, <example> 같은 XML 태그로 감싸 모델이 헷갈리지 않게 한다.
개발자 진입 순서

1·2·9·10(정체성·톤·출력형식·prefill)으로 골격을 먼저 만들어 동작을 확인하고 → 3·4·5(데이터·규칙·예시)를 결과를 보며 반복 개선하세요. 처음부터 10단계를 다 채우지 않아도 됩니다.

01올바른 프롬프팅 기법 5가지개념

구조를 배우기 전에, 작업 성격에 맞는 기법을 고르는 안목이 먼저입니다. 실제 프롬프트 예시는 06 실전 예시에서 자세히 다룹니다.

기법핵심적합한 상황
제로샷 (Zero-Shot)예시 없이 지시문만으로 바로 요청빠르고 단순·명확한 작업
퓨샷 (Few-Shot)입출력 예시를 함께 줘 정확성·일관성 확보형식·컨벤션을 정확히 맞춰야 하는 작업
생각의 사슬 (CoT)작업을 논리적 단계로 분해해 추론 강화디버깅·복잡한 추적 작업
문제 분해 (Decomposition)큰 작업을 작은 단위로 쪼개 순차 진행시스템 설계·다단계 기능 구현
ReAct (Reason+Act)추론 → 행동(탐색·실행) 반복에이전트 자율 탐색·연동 디버깅
실무 감각

단순하면 제로샷으로 시작하고, 정밀도가 필요하면 퓨샷으로 올리세요. 토큰 소모가 곧 비용이므로, 큰 자료는 본문에 붙여넣기보다 파일 첨부로 전달하는 것을 권장합니다. 그 외 Constitutional Prompting, Plan-and-Solve 등도 같은 계열의 확장입니다.

0210단계 구조 한눈에 보기레퍼런스

아래 10개 블록이 시스템 프롬프트의 표준 골격입니다. 색은 다음 섹션의 다이어그램·상세와 1:1로 매칭됩니다.

#스택한 줄 정의필수도

2Tone context말투·태도 설정권장

7Immediate request지금 처리할 실제 요청필수
8Thinking step by step응답 전 추론 유도(CoT)복잡 작업 시

03전체 구조 다이어그램시각화

강의의 Career coach(Joe) 예시를 그대로 옮긴 도식입니다. 좌측 라벨 ↔ 우측 예시 블록의 색이 같으면 같은 스택입니다. (이 SVG는 HTML에 직접 임베드되어 있어 외부 경로 없이도 항상 표시됩니다.)

04각 스택 상세 (1–10)레퍼런스

스택마다 개념 · 무엇을 입력하나 · 예시 문구 · 개발 팁을 정리했습니다.

1. Task context — 역할과 임무

개념. AI에게 누구이고 무엇을 하는가를 가장 먼저 못 박습니다. persona를 부여하면 톤·어휘·판단 기준이 한 방향으로 정렬됩니다.
입력. 정체성 + 목표 + 사용 맥락(누구를 상대로 응답하는가).

You will be acting as an AI career coach named Joe created by the company
AdAstra Careers. Your goal is to give career advice to users. You will be
replying to users who are on the AdAstra site...
개발 팁

코딩 어시스턴트라면 너는 {{기술스택}}에 능숙한 시니어 리뷰어다처럼 역할 + 도메인 + 책임 범위를 한 문장으로.

2. Tone context — 말투

개념. 응답의 정서·격식 수준을 지정. 챗 인터페이스에서 특히 중요합니다.
입력. 한 줄 톤 지시.

You should maintain a friendly customer service tone.
개발 팁

리뷰·분석 작업엔 사실 기반, 간결, 자신감 있게. 리스크 먼저, 권고는 그 다음처럼 순서까지 톤에 담으면 출력 구조가 안정됩니다.

3. Background data — 배경 데이터

개념. AI가 참조할 외부 정보(=context). 지시가 아니라 자료입니다.
입력. 문서·코드 diff·이미지·검색 결과 등. 각 자료를 개별 XML 태그로 감싸 경계를 분리합니다.

Here is the career guidance document you should reference when answering the
user: <guide>{{DOCUMENT}}</guide>
개발 팁

여러 자료는 <diff>…</diff> <tests>…</tests>처럼 태그를 나눠 넣어야 혼동이 없습니다. 큰 데이터는 맨 위보다 이 중간 위치가 권장됩니다.

4. Task description & rules — 규칙

개념. 상호작용의 행동 규칙·금지사항을 리스트로. 모델의 과잉/이탈을 막는 가드레일입니다.
입력. "항상 ~하라", "모르면 ~라고 답하라", "~는 하지 마라" 형태의 명시적 규칙.

Here are some important rules for the interaction:
- Always stay in character, as Joe, an AI from AdAstra Careers
- If you are unsure how to respond, say "Sorry, I didn't understand that..."
- If someone asks something irrelevant, say, "Sorry, I am Joe and I give
  career advice..."
개발 팁

부정형 제약(코드를 임의로 재작성하지 마, 새 엔드포인트 추가 금지)이 특히 효과적입니다. 폴백 응답("모르면 ~")을 미리 정의해 두면 환각을 줄입니다.

5. Examples — 예시 (few-shot)

개념. 말로 설명하는 대신 이상적 입출력을 보여줍니다. 강의에서 "지식 작업에서 가장 효과적인 단일 도구"로 강조합니다.
입력. <example> 태그로 감싼 User→응답 쌍. 엣지 케이스 예시가 가장 가치 높습니다.

Here is an example of how to respond in a standard interaction:
<example>
User: Hi, how were you created and what do you do?
Joe: Hello! My name is Joe, and I was created by AdAstra Careers...
</example>
개발 팁

예시 여러 개면 각각 별도 태그로 감싸고 무엇의 예시인지 라벨을 다세요. 코드 스타일 통일엔 설명보다 예시 1~2개가 빠르고 정확합니다.

6. Conversation history — 대화 히스토리

개념. 직전까지의 멀티턴 맥락 주입. 비어 있을 수 있음을 명시합니다.
입력. <history> 태그 + 동적 변수.

Here is the conversation history... It could be empty if there is no history:
<history> {{HISTORY}} </history>
개발 팁

히스토리가 길면 Context Rot(맥락 누적 노이즈로 정확도 저하)을 유발합니다. 전체를 그대로 넣기보다 요약·슬라이딩 윈도우로 관리하세요.

7. Immediate request — 즉시 요청

개념. 지금 이 순간 처리할 실제 요청. 위의 1~6은 고정 골격, 이 블록만 매 호출 바뀝니다.
입력. <question> 태그 + 동적 변수.

Here is the user's question: <question> {{QUESTION}} </question>
개발 팁

1~6을 CLAUDE.md/시스템 프롬프트의 영구 골격으로, 6·7을 세션 동적 입력으로 분리하면 재사용성이 올라갑니다.

8. Thinking step by step — 추론 유도

개념. 답하기 전에 먼저 생각하게 만드는 Chain-of-Thought.
입력. "먼저 생각하라"는 지시.

How do you respond to the user's question?
Think about your answer first before you respond.
⚠️ 최신 보충 (중요)

Opus 4.x / Sonnet 4.5 등 최신 모델에서는 수동 CoT보다 extended thinking 기능 사용이 공식 권장됩니다. 강의 슬라이드는 이 기능 보편화 이전 표준이므로, 최신 모델에선 extended thinking으로 대체하세요.

9. Output formatting — 출력 형식

개념. 응답의 구조·형식을 강제. 후속 자동 파싱을 위해 필수적입니다.
입력. 형식 지시(XML 태그, JSON 스키마 등).

Put your response in <response></response> tags.
개발 팁

파이프라인에 물릴 출력은 <review>심각도 | 위치 | 설명 | 권고</review>처럼 파싱 가능한 형식을 명시하세요.

10. Prefilled response — 프리필

개념. Assistant 답변의 첫 부분을 미리 채워 형식·태도를 고정하고 군더더기(preamble)를 생략시키는 고급 기법.
입력. Assistant 턴 선두 토큰.

Assistant (prefill)
<response>
개발 팁

<response>까지 미리 넣으면 모델이 곧바로 본문부터 출력합니다. JSON 강제 시 {를 prefill하면 서두 잡설을 막습니다.

05개발자용 변형 — 코드 리뷰 어시스턴트실무

강의 구조를 그대로 실무 프롬프트로 옮긴 예입니다. (대괄호 안 번호 = 위 10단계)

[1 Task]    너는 {{기술스택}}에 능숙한 엄격한 코드 리뷰어다.
[2 Tone]    전문적·간결. 리스크 먼저, 권고는 그 다음.
[3 Data]    <diff>{{이 PR의 git diff}}</diff>
            <tests>{{관련 테스트/CI 로그}}</tests>
[4 Rules]   - 보안 → 성능 → 가독성 → 타입 안정성 순으로 검토
            - 근거 없는 추측 금지. 확인 불가 항목은 "확인 불가"로 표기
            - 코드를 임의로 재작성하지 말 것
[5 Example] <example>
            지적: SQL 문자열 직접 결합 → 인젝션 위험
            권고: 파라미터 바인딩 사용
            </example>
[7 Request] <question>{{리뷰 요청 내용}}</question>
[8 Think]   지적 전에 영향 범위를 먼저 분석하라. (※최신 모델은 extended thinking)
[9 Format]  <review>심각도 | 위치 | 설명 | 권고</review> 표로 출력.
[10 Prefill]<review>
활용 워크플로 (권장 순서)

① 골격 먼저 — 1·2·9·10만 채워 동작 확인 → 템플릿화 · ② 데이터 주입 — 3에 diff·로그를 XML 태그로 분리 · ③ 규칙·예시 반복 개선 — 4·5를 결과 보며 튜닝(엣지 케이스 예시가 최고 ROI) · ④ 분리 배치 — 1·2·4·9는 CLAUDE.md 영구 규칙, 3·6·7은 세션 동적 입력.

06프롬프트 실전 예시 모음실무

기법별로 "상황 → 실제 프롬프트 → 왜 좋은가"를 묶었습니다. 그대로 복사해 쓰고, 본인 작업에 맞게 변수만 바꿔 보세요.

① 제로샷 — 빠른 리팩토링

상황. 의도가 명확하고 단순한 작업. 예시 없이 바로 지시합니다.

이 코드를 OOP 원칙에 맞게 리팩토링하고 테스트케이스도 작성해줘.
왜 좋은가

군더더기 없이 한 문장. 단순 작업에 예시·맥락을 붙이면 토큰만 낭비됩니다. "OOP 원칙", "테스트케이스"처럼 판단 기준을 한 단어씩 박아준 게 포인트.

② 퓨샷 — 스키마 기반 CRUD 생성

상황. 산출물의 형식·컨벤션을 정확히 맞춰야 할 때. 기준이 되는 예시(스키마)를 파일로 첨부해 토큰을 아낍니다.

# support_contacts 테이블 스키마를 파일로 첨부(@table.sql) 후:

@table.sql 을 참고해서 SupportController, service, repository를
코드베이스 컨벤션을 참고해 CRUD로 만들고 Entity도 만들어줘.
컨트롤러에는 프론트에서 호출할 수 있는 HttpClient 호출 스펙을 주석으로 정리해줘.
왜 좋은가

구체적 예시(SQL 스키마)가 출력의 일관성·정확성을 끌어올립니다. "코드베이스 컨벤션 참고"로 기존 스타일에 맞추게 했고, 파일 첨부로 같은 정보를 더 싸게 전달했습니다. 단순하면 제로샷, 정밀도가 필요하면 이렇게 퓨샷으로 올립니다.

③ 생각의 사슬(CoT) — 에러 추적 디버깅

상황. 원인이 한눈에 안 보이는 런타임 에러. 모델이 단계적으로 추적하게 유도합니다.

이 코드 실행 중간에 에러가 발생해. 코드 실행 전후로 로깅을 남기고,
그 로깅을 읽어서 순차적으로 로직이 제대로 된 값을 호출·전달하는지 추적해줘.
에러 로그가 나오면 원인이 뭔지 파악한 뒤 근본적인 해결책을 제시해줘.
왜 좋은가

"로깅 → 읽기 → 순차 추적 → 원인 → 해결"이라는 사고 단계를 명시해 한 번에 답으로 건너뛰지 못하게 했습니다. 최신 모델에선 같은 효과를 extended thinking으로 얻을 수 있습니다.

④ 문제 분해 — 로그인 기능 단계화

상황. 여러 책임이 섞인 기능. 큰 작업을 작은 단위로 쪼개 품질을 올립니다.

로그인 처리는 다음과 같아. 이 로직을 구현해줘.
1) 입력에서 아이디·패스워드를 받는다. 아이디는 이메일 포맷 validation,
   패스워드는 4글자 이상이면 유효로 판단.
2) 서버 전송 시 인증 요청 처리 — 실패면 토스트 메시지, 성공이면 토큰 발급 +
   쿠키 설정 후 응답 반환.
3) 성공 페이지는 xxx로 이동하고, 이동 시 사용자 정보를 조회해 등급·닉네임·랭킹을
   회원 정보 드롭다운에 표시.
왜 좋은가

각 단계의 입력·처리·출력을 분리해 명시하니, 모델이 한 덩어리로 뭉개지 않고 단계별로 정확히 구현합니다. 시스템 설계·연동처럼 범위가 큰 작업에 특히 효과적입니다.

⑤ ReAct — 연동 실패 자율 디버깅

상황. 원인 후보가 많은 외부 연동 실패. 모델이 추론하고 직접 행동(탐색·재시도)하며 좁혀가게 합니다.

RabbitMQ 연동에 계속 실패해. 설정 파일을 분석해서 어디서 연동이 안 되는지 파악하고,
샘플 코드 실행이 안 되면 공식 문서를 참조해 설정 값을 바꿔 다시 테스트해줘.
문제가 발생하면 해당 코드 전후로 로깅을 남기고, 그 로깅을 읽어서
어떤 문제가 있고 어떻게 해결해야 하는지 리포팅해줘.
왜 좋은가

"분석 → 수정 → 재테스트 → 로깅 → 리포팅"의 추론·행동 루프를 명시해, 한 번의 추측이 아니라 탐색하며 수렴하게 만듭니다. 공식 문서 참조까지 허용해 자율 디버깅 범위를 넓혔습니다.

기법 고르는 한 줄 기준

명확·단순 → 제로샷 · 형식 정확성 → 퓨샷 · 추적형 디버깅 → CoT · 범위 큰 구현 → 문제 분해 · 탐색형 연동/에이전트 → ReAct. 그리고 재사용할 프롬프트라면 10단계 구조로 골격을 잡으세요.
