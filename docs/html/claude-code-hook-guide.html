01

훅(Hooks)이란?

⚡ 결정론적 자동화 트리거

훅은 Claude Code 라이프사이클의 특정 시점에 자동 실행되는 사용자 정의 스크립트다. LLM의 판단과 무관하게 매 순간 무조건 실행되는 결정론적 가드레일이라는 점이 핵심이다.

왜 필요한가
CLAUDE.md에 "커밋 전 린트를 돌려라"라고 적어두면 모델이 가끔 잊는다. 훅은 같은 규칙을 시스템 레벨에서 강제한다 — 모델이 거부하든, 잊든, 재해석하든 상관없이 실행된다.

⚖️ 프롬프트(CLAUDE.md) vs 훅(Hooks)

CLAUDE.md / 프롬프트 훅(Hooks)
실행 보장 모델 의존 (probably) 시스템 보장 (always)
판단 주체 LLM (확률적) 셸 / 스크립트 (결정적)
차단 능력 없음 (가이드만) exit 2로 도구 호출 차단
적합한 대상 맥락·뉘앙스 린트·테스트·정책

근거: code.claude.com/docs/en/hooks · ksred.com (2026.02)

첫 훅 만들기

🎯 시나리오Claude가 파일을 저장할 때마다 자동으로 Prettier를 돌려서 포맷을 맞춘다 (PostToolUse + Edit/Write)
1 설정 파일 생성
프로젝트 루트에서
mkdir -p .claude
$EDITOR .claude/settings.json
팀 공유용은 .claude/settings.json,
개인용만은 .claude/settings.local.json
→
2 JSON 작성 
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '.tool_input.file_path' | xargs -r npx prettier --write"
          }
        ]
      }
    ]
  }
}
왜 jq를 쓰나? 공식 문서는 stdin으로 JSON을 전달한다. 환경변수보다 jq 파싱이 표준 패턴이다.

현장에서 가장 많이 쓰는 훅 4종

팀 단위로 공유하기 좋은 베이스라인
RECIPE 01 · 협업 알림 Stop + Notification

💬 Slack으로 작업 완료·대기 알림

장시간 실행되는 Claude 작업이 끝나거나 권한 요청으로 멈췄을 때 Slack으로 즉시 통보. 다른 작업하다가도 놓치지 않는다. Stop은 작업 완료, Notification은 사용자 응답 대기 시 발화. 

#!/bin/bash
# .claude/hooks/slack-notify.sh
INPUT=$(cat)
EVENT=$(echo "$INPUT" | jq -r '.hook_event_name')
PROJECT=$(basename "$(pwd)")

if [ "$EVENT" = "Stop" ]; then
  EMOJI="✅"; TITLE="작업 완료"
else
  MSG=$(echo "$INPUT" | jq -r '.message // ""')
  EMOJI="⏳"; TITLE="입력 대기: $MSG"
fi

curl -s -X POST -H 'Content-Type: application/json' \
  --data "{\"text\":\"$EMOJI $TITLE\",\"attachments\":[{
    \"color\":\"#7c3aed\",
    \"fields\":[
      {\"title\":\"프로젝트\",\"value\":\"$PROJECT\",\"short\":true},
      {\"title\":\"이벤트\",\"value\":\"$EVENT\",\"short\":true}
    ]}]}" \
  "$SLACK_WEBHOOK_URL" > /dev/null
exit 0
설정: Slack에서 Incoming Webhook 생성 후 ~/.zshrc에 export SLACK_WEBHOOK_URL="https://hooks.slack.com/..." 추가. settings.json에 "Stop"과 "Notification" 두 이벤트 모두 등록.
백그라운드로 돌리는 긴 빌드·테스트 작업도 끝난 즉시 Slack에 알림. 채널을 팀원들과 공유하면 동료의 Claude 작업 현황도 볼 수 있어 페어 프로그래밍에 유용.
근거: colombani.ai · Agent SDK · Notification 패턴 · claude-notifications-go
RECIPE 02 · 보안 가드 PreToolUse

🛡️ 파괴적 명령 차단 (rm -rf · git reset --hard)

Claude가 실행하려는 Bash 명령을 검사해 위험 패턴이면 exit 2로 차단. stderr 메시지가 Claude에게 그대로 전달되어 모델이 대안을 찾는다. 

#!/bin/bash
# .claude/hooks/block-destructive.sh
CMD=$(jq -r '.tool_input.command // ""')

if [[ "$CMD" =~ rm[[:space:]]+-rf[[:space:]]+(/|~|\$HOME) ]] \
   || [[ "$CMD" =~ git[[:space:]]+reset[[:space:]]+--hard ]] \
   || [[ "$CMD" =~ curl.*\|.*bash ]]; then
  echo "🚨 차단된 명령: $CMD" >&2
  echo "안전한 대안을 사용하세요." >&2
  exit 2   # 2 = 차단
fi
exit 0
--dangerously-skip-permissions의 위험을 실질적으로 보완. Claude가 차단 사유를 받아 다른 방법을 제안한다.
근거: Anthropic 공식 · stevekinney.com
RECIPE 03 · 커밋 게이트 PreToolUse + Bash

📝 git commit 전 린트·타입체크 강제

Claude가 git commit을 호출하면 인터셉트해서 먼저 린트와 타입체크를 돌린다. 실패 시 차단하고, 모델은 에러 메시지를 받아 자동 수정에 들어간다. 

#!/bin/bash
# .claude/hooks/pre-commit-gate.sh
CMD=$(jq -r '.tool_input.command // ""')

# git commit 명령에만 게이트 적용
if [[ "$CMD" =~ git[[:space:]]+commit ]]; then
  echo "🔍 커밋 전 게이트 실행 중..." >&2

  if ! npx eslint . --max-warnings 0; then
    echo '{"decision":"block","reason":"ESLint 오류. 먼저 수정 후 커밋하세요."}'
    exit 0
  fi

  if ! npx tsc --noEmit; then
    echo '{"decision":"block","reason":"타입 오류. 먼저 수정하세요."}'
    exit 0
  fi
fi
exit 0
왜 PreToolUse + Bash인가? 현재 Claude Code에는 별도의 PreCommit 이벤트가 없다 ( issue #4834 에서 논의 중). PreToolUse + Bash matcher로 커밋 명령을 인터셉트하는 패턴이 현재 표준이다.
모델이 깨진 코드를 절대 커밋하지 못한다. JSON 응답으로 차단 사유가 전달돼 자동 수정 루프가 돈다.
근거: cameronwestland.com · GitHub #4834
RECIPE 04 · 배포 게이트 PreToolUse + Bash

🚀 PR 생성·푸시 전 테스트 강제

gh pr create 또는 git push 전에 테스트 스위트를 돌린다. 실패 시 PR 생성 자체를 차단해 broken PR이 리뷰어에게 가지 않게 막는다. 

#!/bin/bash
# .claude/hooks/pre-pr-gate.sh
CMD=$(jq -r '.tool_input.command // ""')

# gh pr create 또는 git push origin (main 제외) 시 발동
if [[ "$CMD" =~ gh[[:space:]]+pr[[:space:]]+create ]] \
   || [[ "$CMD" =~ git[[:space:]]+push ]]; then

  echo "🧪 PR/푸시 전 테스트 실행..." >&2

  if ! npm test -- --run; then
    echo '{"decision":"block","reason":"테스트 실패. PR 생성 차단."}'
    exit 0
  fi

  # 선택: 커버리지 임계값 검사
  if ! npm run test:coverage -- --check; then
    echo '{"decision":"block","reason":"커버리지 70% 미만"}'
    exit 0
  fi
fi
exit 0
리뷰어 시간 절약. 모델은 테스트가 깨졌다는 사실을 알게 되어 PR 생성 전에 스스로 고친다.
근거: teachmeidea.com · pixelmojo.io
05

고급 — 입력 데이터 · 응답 제어 · 함정

📥 입력 데이터 (stdin) command 타입 기준

Anthropic 공식 사양: 훅 핸들러는 stdin으로 JSON을 받는다. HTTP 핸들러는 같은 JSON이 POST body로 전달된다.

session_id세션 식별자
cwd실행 디렉터리
hook_event_name이벤트명 (PreToolUse 등)
tool_name호출 도구 (Bash, Edit 등)
tool_input.command(Bash) 실행 명령
tool_input.file_path(Edit/Write) 파일 경로
tool_input.content(Write) 작성 내용
tool_response(PostToolUse) 실행 결과

표준 파싱 패턴

INPUT=$(cat)
CMD=$(echo "$INPUT" | jq -r '.tool_input.command // ""')
FILE=$(echo "$INPUT" | jq -r '.tool_input.file_path // ""')
참고: 일부 가이드의 $CLAUDE_FILE_PATHS 같은 환경변수는 비표준이거나 deprecated다. $CLAUDE_PROJECT_DIR은 공식 지원.

📤 응답으로 Claude 제어

훅은 exit code와 stdout JSON 두 가지 방식으로 Claude의 다음 행동을 제어한다.

방식 1 · Exit code

0통과 (계속 진행)
2차단 (PreToolUse만) — stderr가 모델에게 전달됨
기타non-blocking 에러 (사용자에게만 표시)

방식 2 · JSON 응답 (정교)

{
  "decision": "block",
  "reason": "테스트가 실패했습니다. 먼저 수정하세요."
}
JSON 방식의 장점: 차단 사유를 명시적으로 전달할 수 있다. PreToolUse는 "allow" / "deny" / "ask" 결정도 가능.

근거: Agent SDK · hooks

🎓 실무 체크리스트

✅ 권장
  • /hooks로 등록 확인
  • 표준 stdin + jq 파싱
  • 로그를 .claude/logs/에 별도
  • 커밋 전 게이트는 빠르게 (5초 이내)
  • 팀 표준은 프로젝트 스코프에 커밋
  • claude --debug로 검증
❌ 함정
  • 훅에서 또 다른 훅 호출 (무한 루프)
  • 긴 작업 동기 실행 (전체 테스트 등)
  • 비밀번호·토큰 stdout 출력 금지
  • 상대 경로 의존 (cwd가 다를 수 있음)
  • 팀 합의 없이 강제 차단 정책 추가
  • 샌드박스 없음 — 풀 사용자 권한 실행
⚠️ 보안 경고: Anthropic 공식 문서가 명시: 훅은 샌드박스 없이 사용자 전체 권한으로 실행된다. 잘못 짠 훅 하나가 파일을 지우거나 비밀을 유출시킬 수 있으니 코드 리뷰 필수.
