먼저 알아야 할 한 가지

GitHub에 코드를 올릴 때(git push) 입력하는 건 GitHub 로그인 비밀번호가 아닙니다. 2021년 8월부터 비밀번호 인증이 폐지되어, 대신 브라우저 로그인 또는 토큰(PAT)으로 인증합니다. 비개발자라면 처음 푸시할 때 자동으로 뜨는 브라우저 로그인 창만 따라가면 됩니다. 자세한 건 2단계에서 다룹니다.

0시작 전에 — Git과 GitHub의 차이난이도 쉬움

두 단어가 비슷해 헷갈리지만 역할이 다릅니다.

구분무엇인가비유
Git내 컴퓨터에서 파일의 변경 이력을 기록·관리하는 프로그램저장 버튼이 달린 "기록 노트"
GitHub그 기록을 인터넷에 올려 두고 여럿이 함께 보는 공간(웹 서비스)노트를 올려 두는 "공용 책장"

즉 Git은 내 컴퓨터(로컬)에서 돌아가고, GitHub은 인터넷(원격)에 있습니다. 우리는 로컬에서 작업한 기록을 GitHub으로 올리고(push), 남이 올린 변경을 내려받습니다(pull).

1GitHub 가입하기난이도 쉬움

https://github.com/signup 에 접속해 순서대로 입력합니다.

  1. Email — 실제 사용하는 이메일(인증 메일이 옵니다).
  2. Password — GitHub 로그인용 비밀번호. (뒤에서 설명하지만, 이 비밀번호는 코드를 올릴 때는 쓰지 않습니다.)
  3. Username — 전 세계에서 고유한 아이디. 저장소 주소에 들어갑니다. 예: github.com/내아이디/저장소
  4. 이메일로 받은 인증 코드 입력 → 가입 완료.
[스크린샷 영역] GitHub 가입 화면 — Email / Password / Username 입력란과 [Create account] 버튼
2단계 인증(2FA)은 사실상 필수

2023년부터 GitHub은 코드를 올리는 모든 사용자에게 2단계 인증(2FA)을 의무화했습니다. 가입 후 또는 푸시를 시작하면 2FA 설정 안내가 뜹니다. 인증 앱(Google Authenticator, Microsoft Authenticator 등)을 스마트폰에 설치해 등록하는 방식을 권장합니다. 등록 시 보여주는 복구 코드(recovery codes)는 반드시 따로 저장하세요. 휴대폰을 잃어버렸을 때 계정을 되살리는 유일한 열쇠입니다.

2"비밀번호"의 진실 — 푸시 인증의 원리난이도 기본

비개발자가 가장 많이 막히는 지점입니다. git push를 했더니 비밀번호를 물어봐서 GitHub 로그인 비밀번호를 넣으면 이런 에러가 납니다.

remote: Support for password authentication was removed on August 13, 2021.
fatal: Authentication failed for 'https://github.com/USER/REPO.git/'

원인: GitHub은 2021년 8월 13일에 비밀번호로 코드 작업을 인증하는 방식을 폐지했습니다. 대신 다음 셋 중 하나로 인증합니다. 비개발자에게 쉬운 순서로 정리했습니다.

방법 A. 브라우저 로그인 (가장 쉬움 · 권장)

Windows용 Git에는 Git Credential Manager가 기본 포함되어 있습니다(맥은 설치 가능). 이 경우 처음 푸시할 때 브라우저 창이 저절로 떠서 GitHub에 로그인하면 끝납니다. 토큰을 직접 만질 필요가 없고, 그 다음부터는 아무것도 묻지 않습니다.

[스크린샷 영역] 첫 push 시 자동으로 뜨는 "Sign in to GitHub" 브라우저 창 → [Authorize] 클릭

방법 B. gh CLI 로 한 번에

GitHub 공식 도구 gh를 설치했다면 명령 한 줄로 인증·설정이 모두 끝납니다.

$ gh auth login
# GitHub.com → HTTPS → "Login with a web browser" 선택
# 터미널에 표시된 8자리 코드를 브라우저에 입력하면 완료

이 한 번으로 토큰 발급·저장·git 연결이 자동 처리됩니다.

방법 C. 토큰(PAT) 직접 발급 (수동)

위 방법을 못 쓰는 환경이거나 외부 도구에 토큰을 직접 넣어야 할 때 씁니다.

GitHub 우상단 프로필 → Settings → Developer settings
  → Personal access tokens → Tokens (classic)
  → Generate new token (classic)
  → Expiration: 90 days,  Scopes: repo 체크 → Generate

발급된 ghp_… 토큰은 그 화면을 벗어나면 다시 볼 수 없으니 바로 복사해 안전한 곳에 보관합니다. 이후 푸시할 때 Username엔 GitHub 아이디, Password 자리엔 이 토큰을 붙여넣습니다.

$ git push origin main
Username for 'https://github.com': 내아이디
Password for 'https://...github.com': ghp_여기에토큰붙여넣기
기억할 것

① 코드 작업 인증에 GitHub 로그인 비밀번호는 쓰이지 않는다. ② 비개발자는 방법 A(브라우저 로그인)가 가장 편하다. ③ 토큰은 비밀번호처럼 다루고, 1년간 안 쓰면 GitHub이 자동으로 폐기한다.

3내 컴퓨터에 Git 준비하기난이도 기본

먼저 Git이 설치돼 있는지 확인합니다. 버전이 출력되면 설치된 것입니다.

$ git --version
git version 2.43.0

설치돼 있지 않다면 https://git-scm.com/downloads 에서 운영체제에 맞게 설치합니다.

내 정보(로컬 계정) 등록하기

커밋 기록에 "누가 변경했는지" 남기기 위해 이름과 이메일을 한 번 등록합니다. 이메일은 GitHub 가입 이메일과 같게 맞추면 GitHub에서 내 활동으로 인식됩니다.

$ git config --global user.name "Hong Gildong"
$ git config --global user.email "내가입이메일@example.com"
$ git config --global init.defaultBranch main   # 기본 브랜치 이름을 main으로

제대로 등록됐는지는 아래로 확인합니다.

$ git config --list
"로컬 계정 변경"이란?

회사·개인 등 이메일을 바꿔 커밋하고 싶으면 같은 명령으로 user.email을 다시 설정하면 됩니다. 특정 프로젝트에서만 다르게 쓰려면 그 폴더 안에서 --global을 빼고 실행하세요.

4첫 리포지토리 만들기난이도 쉬움

리포지토리(repository, 줄여서 repo)는 프로젝트 하나가 담기는 저장소 폴더입니다. GitHub 웹에서 만듭니다.

  1. GitHub 우상단 + → New repository.
  2. Repository name 입력 (예: my-first-repo).
  3. 공개 범위 선택: Public(누구나 봄) / Private(나/허락한 사람만).
  4. Add a README file 체크 — 첫 파일이 생겨 바로 클론하기 편합니다.
  5. Create repository 클릭.
[스크린샷 영역] New repository 화면 — Repository name / Public·Private / Add a README 체크박스 / [Create repository]

만들어진 저장소 주소는 다음 형태입니다. 다음 단계에서 이 주소를 씁니다.

https://github.com/내아이디/my-first-repo.git

5클론 → 커밋 → 푸시 기본난이도 기본

가장 많이 쓰는 핵심 흐름입니다. 원격 저장소를 내 컴퓨터로 복제(clone)한 뒤, 파일을 고치고, 저장 지점을 만들고(commit), 올립니다(push).

① 클론 — 내 컴퓨터로 가져오기

$ git clone https://github.com/내아이디/my-first-repo.git
$ cd my-first-repo

② 파일 수정 후 상태 확인

README 파일을 메모장이나 편집기로 고친 뒤, 무엇이 바뀌었는지 봅니다.

$ git status
# modified: README.md  ← 빨갛게 표시되면 "아직 저장 안 됨"

③ 커밋 — 저장 지점 만들기

add는 "이 변경을 포함하겠다"는 표시, commit은 "여기까지를 한 묶음으로 저장"입니다.

$ git add .                          # 바뀐 파일 전부 포함 (특정 파일만: git add README.md)
$ git commit -m "README에 프로젝트 소개 추가"

④ 푸시 — GitHub에 올리기

$ git push

처음 푸시할 때만 2단계에서 설명한 브라우저 로그인 창이 뜹니다. 한 번 인증하면 이후엔 묻지 않습니다.

README 없이 빈 저장소로 시작했다면

내 폴더를 처음부터 연결하는 경우엔 아래처럼 합니다.

$ git init
$ git add .
$ git commit -m "first commit"
$ git branch -M main
$ git remote add origin https://github.com/내아이디/my-first-repo.git
$ git push -u origin main          # -u: 다음부터 git push만 해도 됨

6브랜치로 작업하고 PR로 합치기난이도 중급

브랜치(branch)는 본래 작업을 건드리지 않고 따로 떼어 안전하게 작업하는 작업 줄기입니다. 보통 main(배포본)과 develop(개발 통합본)을 두고, 새 작업은 거기서 가지를 칩니다.

① 새 브랜치 만들고 그 위에서 작업

$ git switch -c feature/login     # 생성+이동 (옛 방식: git checkout -b feature/login)
# ... 파일 작업 ...
$ git add .
$ git commit -m "feat: 로그인 화면 추가"

② 브랜치를 원격에 올리기

$ git push -u origin feature/login

③ PR(Pull Request) 만들기

PR은 "내 브랜치의 변경을 develop에 합쳐 달라"는 합치기 요청 + 리뷰 공간입니다. 푸시 후 GitHub 저장소 페이지 상단에 Compare & pull request 버튼이 나타납니다.

[스크린샷 영역] 저장소 상단의 "Compare & pull request" 노란 배너 → PR 작성 화면(base: develop ← compare: feature/login)
  1. base를 develop, compare를 feature/login으로 지정.
  2. 제목·설명 입력 → Create pull request.
  3. 리뷰·확인 후 Merge pull request로 develop에 합쳐집니다.

명령으로 PR을 만들려면 gh를 씁니다.

$ gh pr create --base develop --head feature/login \
     --title "로그인 화면 추가" --body "로그인 UI와 검증 추가"
PR 없이 로컬에서 직접 develop에 합치려면

혼자 연습할 때는 PR 대신 로컬 병합도 가능합니다. (협업에서는 PR을 권장)

$ git switch develop
$ git pull                # 원격 develop 최신화
$ git merge feature/login
$ git push

7fetch · pull · push 구분하기난이도 중급

세 명령은 모두 "원격과 주고받기"지만 방향과 동작이 다릅니다.

명령방향하는 일
git fetch원격 → 로컬원격의 변경을 가져오기만 함. 내 작업 파일은 그대로 두고, 무엇이 바뀌었는지 먼저 확인할 때.
git pull원격 → 로컬fetch + merge. 가져온 변경을 현재 브랜치에 바로 합침.
git push로컬 → 원격내 커밋을 원격에 올림.
 
$ git fetch                       # 가져오기만 (안전, 미리보기)
$ git log origin/develop --oneline # 원격에 뭐가 들어왔는지 확인
$ git pull                        # 확인 후 합치기
$ git push                        # 내 작업 올리기
실무 습관

작업을 시작하기 전 git pull로 최신 상태를 먼저 받아 두면, 나중에 충돌이 크게 나는 걸 줄일 수 있습니다.

8merge 이해하기난이도 중급

merge는 한 브랜치의 변경을 다른 브랜치로 합치는 작업입니다. 합치는 쪽 브랜치로 이동한 뒤 합칠 브랜치를 지정합니다.

$ git switch develop          # 받는 쪽으로 이동
$ git merge feature/login     # feature/login의 변경을 develop으로

두 가지 결과가 있습니다.

  • Fast-forward — develop에 그동안 새 커밋이 없으면, 그냥 포인터만 앞으로 옮겨 깔끔히 합쳐집니다.
  • 3-way merge — 양쪽 모두 변경이 있으면 Git이 둘을 합친 병합 커밋을 만듭니다. 이때 같은 줄을 서로 다르게 고쳤다면 충돌(conflict)이 납니다(다음 단계).

9충돌(conflict) 해결 기본난이도 중급

같은 파일의 같은 줄을 두 브랜치가 서로 다르게 고치면 Git이 자동으로 못 합치고 충돌을 알립니다.

$ git merge feature/login
Auto-merging README.md
CONFLICT (content): Merge conflict in README.md
Automatic merge failed; fix conflicts and then commit the result.

① 충돌 표시 읽기

충돌난 파일을 열면 이런 표시가 들어가 있습니다.

<<<<<<< HEAD
현재 브랜치(develop)의 내용
=======
들어오는 브랜치(feature/login)의 내용
>>>>>>> feature/login
  • <<<<<<< HEAD ~ ======= : 지금 내 브랜치 내용
  • ======= ~ >>>>>>> : 합쳐 들어오는 내용

② 직접 고치기

최종적으로 남길 내용만 남기고 표시줄 3개(<<<, ===, >>>)를 모두 지웁니다. 둘 다 살릴 수도, 하나만 택할 수도, 새로 합쳐 쓸 수도 있습니다.

③ 해결 표시 후 커밋

$ git add README.md       # 해결했다고 표시
$ git commit               # 병합 커밋 생성 (메시지 자동 제안됨)
$ git push
막막하면 일단 멈추기

충돌이 너무 복잡해 손대기 두렵다면 git merge --abort로 병합 직전 상태로 되돌릴 수 있습니다. 안전하게 다시 시작하세요.

10합쳐지지 않을 때 트러블슈팅난이도 중급

자주 만나는 상황과 기본 해결법입니다. 막히면 먼저 git status가 알려주는 안내 문구를 읽는 습관이 가장 중요합니다.

증상 1. push가 거부됨 (rejected)

 ! [rejected]        main -> main (fetch first)
error: failed to push some refs to 'https://github.com/...'

원인: 내가 올리는 사이에 원격에 다른 변경이 먼저 들어왔습니다. 해결: 원격 변경을 먼저 받아 합친 뒤 다시 올립니다.

$ git pull        # 원격 변경 합치기 (충돌 나면 9단계대로 해결)
$ git push

증상 2. 병합을 취소하고 싶음

$ git merge --abort   # 병합 시작 전 상태로 복귀

증상 3. 엉뚱한 브랜치에서 작업함

$ git switch 올바른브랜치     # 아직 커밋 안 했다면 변경이 따라 이동
# 변경을 잠깐 치워 두고 이동하려면:
$ git stash               # 작업 임시 보관
$ git switch 올바른브랜치
$ git stash pop           # 보관한 작업 꺼내기

증상 4. 인증 실패로 push가 안 됨

2단계의 인증 문제입니다. 브라우저 인증을 다시 하거나(gh auth login), 토큰을 새로 발급해 입력합니다. 이전에 잘못 저장된 인증 정보가 캐시돼 있으면 OS의 자격 증명 관리자(Windows) · 키체인(macOS)에서 github.com 항목을 지운 뒤 다시 인증하세요.

증상 5. 현재 상태가 헷갈림

$ git status                       # 지금 무슨 상태인지 (가장 먼저!)
$ git remote -v                     # 연결된 원격 주소 확인
$ git log --oneline --graph -10     # 최근 이력을 그림으로
트러블슈팅 3원칙

① git status를 먼저 읽는다 — Git이 해야 할 일을 거의 알려준다. ② 되돌릴 수 있는 명령(--abort, stash)부터 쓴다. ③ 강제 명령(push --force, reset --hard)은 함부로 쓰지 않는다 — 남의 작업이나 내 기록이 사라질 수 있다.
