# ANALYSIS — Lucky Draw 확정 결정 · 변경 이력

> **역할**: 추가 요건을 반영해 확정한 **설계 결정**과, 그 결정이 기존 문서를 어떻게 바꿨는지의 **변경 이력**을 기록한다.
> 상세 명세는 중복하지 않는다 — 데이터·알고리즘은 `TRD.md`, 기능·검증은 `REQUIREMENTS.md`, 목적·범위는 `PRD.md`.
> 정본은 구현(`lucky-draw.html`). 본 문서·PRD·TRD·REQUIREMENTS는 그 구현과 1:1로 정합한다.

---

## 1. 확정 결정 (Source of Decisions)

| 항목 | 결정 |
|---|---|
| 추첨 단계 | **2단계**: ① 번호 추첨(즉시 공개) → ② 경품 원판 회전 |
| 입력 | **인원수(N) + 경품(이름·갯수)** 2종 |
| 번호 추첨 | `1..N` **균등 랜덤**, **중복 불가(1인 1회)**, ~0.8초 공개 |
| 원판 회전 | 경품용 원판 1개, TRD §5.4 알고리즘 그대로(6~8바퀴, ±35% jitter) |
| 섹션 구성 | **경품 갯수 합 + 꽝 1칸**의 고정 길이 S |
| 재고 소진 | **섹션 수 유지 + 재배정**(재고 최다 경품으로, 동률 등록순, 없으면 꽝) |
| 꽝 | **항상 1칸 상시 존재** (인원이 많아도 재미 유지) |
| 꽝 색상 | **중립 회색 `#D4D4D4`** (액센트 빨강은 슬라이스 금지 규칙 우선) |
| 컨페티 | **당첨만** 발사 (꽝 제외) |
| 종료 | `뽑은 번호 수 === N` 이면 SPIN 비활성 |

> 미결 기본값(이의 시 조정): 꽝 1칸 고정 · 재배정 대상=재고 최다 · 컨페티 당첨만 · 인원/경품 수 사전 제약 없음.

## 2. 변경 이력 — 기존 문서를 덮어쓴 부분

| # | 쟁점 | 기존(초안) | 확정(현재 구현) |
|---|---|---|---|
| C1 | 추첨 플로우 | 경품 원판 1단계만 | **번호 추첨 → 경품 원판 2단계** |
| C2 | 입력 항목 | 경품(이름+갯수)만 | **인원수 + 경품 2종** |
| C3 | 재고 0 처리 | "휠에서 제거"(슬라이스 사라짐) | **섹션 유지 + 재배정** |
| C4 | 꽝 | 없음 | **꽝 1칸 상시 (회색)** |
| C5 | 경품 데이터 | `{id,name,count,color}` | **`{id,name,count}`** — 원판이 2색 교차라 경품별 색 불필요 |
| C6 | 경품 소진 후 | "SPIN 비활성 + 경품 추가 안내" | **꽝으로 N까지 진행** 후 종료 |
| C7 | 예시 데이터 | "… 꽝 4" 경품 포함 | **꽝 제외**(자동 1칸) + 인원 10 추가 |
| C8 | 리스트 vs 휠(count 0) | "삭제 vs 회색" 모순 | 휠은 재배정(삭제 안 함) / 리스트는 회색 유지 |

**유지(충돌 없음)**: 단일 파일·무의존성, SVG 원판 + Canvas 컨페티, 회전 5.5s `cubic-bezier(0.17,0.67,0.12,0.99)`, TRD §5.4 정지 각도, 흑백+빨강·AI 흔적 배제, `escapeHtml`, 접근성·`prefers-reduced-motion`, 100KB 이하.

## 3. 핵심 불변식 (구현·검증 기준)

- ⭐ **원판 정지 슬라이스(`winIdx`) = 결과 패널 = 모달 = 기록** — 4곳 일치. (시뮬레이션 S=2·5·9·18·37 각 5000회, 편차 ≤ 슬라이스 반각 → 통과)
- **섹션 수 불변** — 재배정은 라벨 교체만, 길이 변경 금지. (예시 세션 10회 추첨 동안 S=9 유지 확인)
- **번호 무중복·종료** — `1..N` 정확히 N회로 소진, 추가 추첨 시 `null`.

## 5. 알려진 한계 / 후속

- 삭제(×) 버튼 36px — 주요 버튼은 ≥44px이나 이 컨트롤만 44 미만(밀도 고려). 필요 시 상향.
- 재배정은 **확률이 잔여 재고와 정확히 비례하지 않음**(섹션 고정의 트레이드오프, 소진 시 자기보정). 엄밀 비례가 필요하면 "남은 재고로 원판 재구성" 방식으로 전환.
- 다중 꽝·확률 가중치 직접 입력 등은 `REQUIREMENTS.md` §8 확장 포인트 참조.
