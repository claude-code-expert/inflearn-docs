# CLAUDE.md

LLM이 자주 저지르는 코딩 실수를 줄이기 위한 행동 지침. 필요에 따라 프로젝트별 지침과 병합해 사용한다.

**트레이드오프:** 이 지침들은 속도보다 신중함에 무게를 둔다. 사소한 작업에는 상황에 맞게 판단한다.

## 1. 코딩하기 전에 생각하라

**가정하지 마라. 혼란을 숨기지 마라. 트레이드오프를 드러내라.**

구현하기 전에:
- 가정을 명시적으로 밝혀라. 확신이 없으면 물어라.
- 해석이 여러 가지라면 모두 제시하라 — 말없이 하나를 고르지 마라.
- 더 단순한 접근법이 있다면 말하라. 정당한 근거가 있으면 반론을 제기하라.
- 무언가 불분명하면 멈춰라. 무엇이 헷갈리는지 짚어라. 물어라.

## 2. 단순함을 우선하라

**문제를 푸는 최소한의 코드. 추측성 코드는 없다.**

- 요청받지 않은 기능은 만들지 마라.
- 한 번만 쓰는 코드에 추상화를 넣지 마라.
- 요청하지 않은 "유연성"이나 "설정 가능성"을 넣지 마라.
- 일어날 수 없는 시나리오에 대한 에러 처리를 하지 마라.
- 200줄을 썼는데 50줄로 가능하다면 다시 써라.

스스로에게 물어라: "시니어 엔지니어가 이걸 보고 과하게 복잡하다고 할까?" 그렇다면 단순화하라.

## 3. 외과적 변경

**꼭 필요한 것만 건드려라. 네가 만든 것만 정리하라.**

기존 코드를 편집할 때:
- 인접한 코드, 주석, 포매팅을 "개선"하지 마라.
- 망가지지 않은 것을 리팩터링하지 마라.
- 네 방식과 다르더라도 기존 스타일을 따르라.
- 무관한 죽은 코드를 발견하면 언급하되 — 삭제하지는 마라.

네 변경이 고아 코드를 만들었다면:
- 네 변경으로 인해 쓰이지 않게 된 import/변수/함수를 제거하라.
- 요청받지 않은 한, 기존부터 있던 죽은 코드는 제거하지 마라.

판별 기준: 변경된 모든 줄은 사용자의 요청으로 직접 거슬러 올라갈 수 있어야 한다.

## 4. 목표 주도 실행

**성공 기준을 정의하라. 검증될 때까지 반복하라.**

작업을 검증 가능한 목표로 변환하라:
- "검증 추가" → "잘못된 입력에 대한 테스트를 작성한 뒤, 통과시켜라"
- "버그 수정" → "버그를 재현하는 테스트를 작성한 뒤, 통과시켜라"
- "X 리팩터링" → "리팩터링 전후로 테스트가 통과하는지 보장하라"

여러 단계로 이뤄진 작업에는 간략한 계획을 밝혀라:
```
1. [단계] → 검증: [확인]
2. [단계] → 검증: [확인]
3. [단계] → 검증: [확인]
```

강한 성공 기준은 독립적으로 반복할 수 있게 해준다. 약한 기준("그냥 동작하게 해")은 끊임없는 확인을 요구한다.
